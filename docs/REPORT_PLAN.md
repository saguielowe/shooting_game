## 4. 设计取舍（报告重点）

### 4.1 为什么这么选
- AI先用状态机而不是行为树
  - 原因：实现成本低、可调试性高、课程项目周期内收益更高。
- 战斗与碰撞采用自定义轻量规则，而不是完整物理引擎
  - 原因：行为可控、平衡调参快、实现复杂度可控。
- 词条系统采用枚举+状态容器，而不是脚本DSL
  - 原因：上手快、代码路径清晰、迭代风险低。
- UI使用Qt Widgets，而不是重迁移到QML/游戏引擎UI
  - 原因：与现有工程一致，联调和维护成本最低。

### 4.2 承认有更优方案但暂不采用
- 行为树/EQS可提升AI上限，但当前阶段投入产出比不高。
- 物理引擎可提升真实感，但会显著增加调参与联调成本。
- 数据驱动脚本化可提升扩展性，但会增加系统复杂度与调试负担。
